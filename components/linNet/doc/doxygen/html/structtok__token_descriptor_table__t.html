Detailed Description

See other type tok_tokenDescriptor_t: Here we have the complete token definition table, which contains all client specified symbols and their token values.

Field Documentation

unsigned int tok_tokenDescriptorTable_t::noTokenDescriptions

The number of client defined symbols, or the number of elements of tokenDescriptorAry. May be 0 if no custom symbols are required.

const tok_tokenDescriptor_t* tok_tokenDescriptorTable_t::tokenDescriptorAry

The list of all client defined symbols, organized as an array. There's no specific sort order for the array.

Remarks
: No symbol must be NULL or the empty string and no two symbols must be the identical. The latter condition is not checked, the former are in DEBUG compilation (by assertion).
Using identical token values (or type) is valid and supported. Different character sequences would be reported in a transparent way to the client as same token. One could e.g. define the symbol ";" as type tok_tokenTypeEndOfLine, in which case the semicolon would have the same meaning to the client parser as an end of line character.
const char* tok_tokenDescriptorTable_t::startComment

The tokenizer supports two comment styles. The first one uses two symbols to mark beginnning and end of the comment (both including). The second one uses a symbol to mark the beginning of the comment (including) and the comment ends at the next end of line character (excluding).
Here the client defines the beginning of a comment of the first style.
If no such comment is needed, the value is set to NULL.
